# Nornicel R&D Knowledge Graph

**Единая карта знаний R&D для горно-металлургической отрасли.**

Система связывает разнородные сущности R&D (публикации, эксперименты, технологические
решения, материалы, оборудование, эксперты, выводы) в граф знаний и отвечает на сложные
многопараметрические запросы на естественном языке — с указанием источника, уровня
достоверности и географии (отечественная / зарубежная практика).

> Кейс хакатона: устранить потерю институциональной памяти и дублирование R&D-усилий
> в горно-металлургических исследованиях (обессоливание воды, электроэкстракция никеля,
> распределение драгметаллов между штейном и шлаком, закачка шахтных вод и т.д.).

---

## Что умеет система

- **Импорт и извлечение знаний** из статей, отчётов, патентов (RU/EN) — LLM-пайплайн
  вытаскивает сущности, связи и **числовые ограничения** («сульфаты ≤ 300 мг/л», «климат: холодный»).
- **Граф знаний** в Neo4j с доменной онтологией и **моделью верификации** (источник,
  уровень достоверности, дата актуализации, число подтверждений).
- **Запросы на естественном языке** → Cypher (GraphRAG): многоуровневая фильтрация по
  материалу, процессу, условиям, географии и временному диапазону.
- **Синтез ответов** уровня «литературный обзор»: группировка по методу/году/географии,
  выделение консенсуса и зон разногласий, подсветка **пробелов в знаниях**.
- **Различение отечественной и зарубежной практики** через фильтр по географии.

### Целевые запросы (демо-сценарии)

1. «Какие методы обессоливания воды подходят, если исходная вода содержит сульфаты, хлориды,
   Ca, Mg, Na по 200–300 мг/л, а требуемый сухой остаток ≤ 1000 мг/дм³?»
2. «Какие технические решения циркуляции католита при электроэкстракции никеля описаны в
   мировой практике и какая скорость потока считается оптимальной?»
3. «Покажите все эксперименты и публикации по распределению Au, Ag и МПГ между медным/
   никелевым штейном и шлаком за последние 5 лет.»
4. «Какие способы закачки шахтных вод в глубокие горизонты применялись в России и за рубежом,
   и каковы их технико-экономические показатели?»

## Архитектура (кратко)

```
Документы (RU/EN)                    Запрос на естественном языке
  │  PDF / DOCX / TXT                     │
  ▼                                       ▼
Ingest-пайплайн                     Query-пайплайн (GraphRAG)
  • разбиение на чанки                 • NL → Cypher (LLM)
  • LLM-извлечение сущностей/связей    • обход графа + числовые фильтры
  • нормализация (синонимы, единицы)   • семантический поиск по эмбеддингам
  • разметка достоверности/источника   • синтез ответа + цитаты
  │                                       │
  ▼                                       ▼
        ┌─────────────────────────────────────────┐
        │   Neo4j — граф знаний + vector index     │
        │   Онтология: Material, Process, ...      │
        └─────────────────────────────────────────┘
                          ▲
                          │ REST API (Django + DRF)
                          ▼
                  Frontend (React + граф-визуализация)
```

Онтология, модель верификации и примеры Cypher описаны в исходном коде:
`backend/knowledge/ontology.py`, `data/ontology/` и `backend/knowledge/graph/schema.cypher`.

## Технологический стек и обоснование

| Слой | Выбор | Почему |
|------|-------|--------|
| Граф-БД | **Neo4j** (Aura Free cloud) | Рекомендован кейсом; Cypher нагляден для демо; встроенный vector index для семантики; облако = без локальной установки Docker |
| Бэкенд | **Django + Django REST Framework** | Быстрый REST, готовые аутентификация/ролевая модель/админка/аудит; Python-экосистема рядом с рекомендованными NLP-инструментами |
| Реляционная БД | **SQLite → Postgres** | Django ORM для пользователей, ролей и **аудита** (требование ИБ); граф знаний — отдельно в Neo4j |
| Извлечение (NLP) | **LLM (YandexGPT)**, интерфейс под spaCy/ruBERT | Качество извлечения сущностей/чисел за 2 дня выше рукописного; мультиязычность RU/EN «из коробки» |
| NL → Cypher | **LLM + схема онтологии** | Ключевая фича демо: запрос на естественном языке без знания Cypher |
| Фронтенд | **React + Vite + react-force-graph** | Визуализация цепочек «материал → процесс → оборудование → результат» |

**Архитектурное разделение.** Django ORM (SQLite/Postgres) держит реляционную часть —
пользователи, роли, **аудит действий** (`knowledge/models.py::QueryLog`). Граф знаний живёт
в Neo4j и доступен через сервисный слой `knowledge/services/graph.py`. Так ролевой доступ,
логирование и админка достаются «из коробки» Django, а обход связей и числовые фильтры —
сильными сторонами графовой БД.

**Осознанный компромисс по NLP.** Кейс рекомендует DeepPavlov / spaCy / ruBERT. Мы выбираем
LLM-извлечение как основной путь ради скорости и качества за 2 дня, но слой извлечения
(`knowledge/services/extractor.py`) спроектирован так, что офлайн spaCy/ruBERT-реализацию
можно подключить как альтернативу без переписывания графа и API.

## Быстрый старт

> Python в системе доступен через launcher `py` (алиасы `python`/`python3` могут вести на
> заглушку Microsoft Store). Ниже — команды через `py`.

```bash
# 1. Конфигурация
cp .env.example .env            # заполнить NEO4J_* и LLM_API_KEY (YandexGPT)

# 2. Виртуальное окружение и зависимости
cd backend
py -m venv .venv
.venv/Scripts/activate          # Windows (PowerShell: .venv\Scripts\Activate.ps1)
pip install -r requirements.txt

# 3. Реляционные миграции Django (пользователи/роли/аудит)
py manage.py migrate

# 4. Инициализация схемы графа + демо-данные (Neo4j должен быть запущен)
py manage.py graph_init
py manage.py seed_demo          # связный демо-граф для сквозных запросов

# 5. Запуск API
py manage.py runserver          # http://localhost:8000/api/health

# --- Frontend (отдельный терминал) ---
cd ../frontend
npm install && npm run dev      # http://localhost:5173
```

> **Neo4j:** можно использовать Aura Free (облако) или локальный сервер. Готовые скрипты
> запуска — в `scripts/` (`start-neo4j.ps1`, `start-backend.ps1`, `start-frontend.ps1`).

> **Данные кейса** (Яндекс.Диск: https://disk.yandex.ru/d/npigiuw4Rbe9Pg) не входят в репозиторий.
> Положите распакованный корпус в `data/corpus/` и запустите `py manage.py ingest --source ../data/corpus`.

```
│   │   ├── ontology.py         # типы сущностей/связей + Provenance

│   │   ├── services/           # graph, extractor, normalize, nl2cypher

│   │   └── management/commands # graph_init, ingest
│   └── requirements.txt
├── frontend/                   # React + Vite (визуализация графа и поиск)
├── data/
│   └── ontology/               # типы сущностей, связей, синонимы RU/EN, единицы

└── README.md
```
